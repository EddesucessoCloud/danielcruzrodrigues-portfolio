1. Diagnose Before You Tune

The most common mistake in Spark tuning is changing configuration blindly. Before touching a single parameter, open the Spark UI and read it. Every performance problem leaves a distinct signature — once you recognise the patterns, tuning becomes systematic rather than superstitious.

I've seen teams spend days tweaking spark.sql.shuffle.partitions when their real problem was a broadcast join hitting a 10 GB lookup table. The Spark UI would have shown them a 40-minute sort-merge join stage in 30 seconds. Read the UI first.

Key Spark UI signals

What you seeWhat it meansWhere to look
One task 10× slower than others Data skew — one partition is disproportionately large Stages → task duration distribution
Spill to disk > 0 Executor memory too small for the shuffle/aggregation Stages → "Shuffle Spill (Memory)" column
GC time > 10% of task time Too many objects in heap; executor overhead too low Executors tab → "GC Time"
Thousands of tiny tasks Over-partitioned data — too many small files Stages → task count vs data size
Long "Getting Result" phase Driver collecting too much data (large .collect()) Jobs → event timeline
Spark History Server on AWS: On EMR, the Spark History Server is available at port 18080 on the primary node. On Databricks, it's under Cluster → Spark UI. On AWS Glue, enable job metrics and use the CloudWatch dashboard — the Spark UI isn't directly accessible, but Glue exposes shuffle read/write bytes and heap usage as metrics.

2. Executor Sizing: The Fat vs Thin Executor Debate

How you partition vCPUs and memory across executors has a larger impact on performance than most configuration parameters. There are two schools of thought: fat executors (few executors, many cores each) and thin executors (many executors, few cores each).

The 4-core rule

In practice, 4–5 cores per executor is the sweet spot for most batch ETL workloads. More than 5 cores per executor causes HDFS throughput degradation (HDFS client threads compete at the file system level). Fewer than 3 cores wastes JVM overhead — each executor has a fixed per-process cost. 

# Good: 4 cores, ~14 GB executor memory on a 16-core, 64 GB node
# Leaves 1 core + memory for OS, YARN NM, and off-heap buffers

# Formula: executors_per_node = floor((node_cores - 1) / executor_cores)
#          executor_memory    = floor((node_memory - OS_overhead) / executors_per_node) * 0.9

# For r6i.4xlarge (16 vCPU, 128 GB):
#   executors_per_node = floor((16 - 1) / 4) = 3
#   executor_memory    = floor((128 - 2) / 3) * 0.9 ≈ 37 GB
#   memoryOverhead     = max(384 MB, 0.1 * 37 GB) ≈ 4 GB

spark.executor.cores         4
spark.executor.memory        37g
spark.executor.memoryOverhead 4g
spark.driver.memory          8g
spark.driver.maxResultSize   4g
memoryOverhead is not optional: Overhead covers off-heap memory — PySpark's Python worker processes, native Arrow buffers, and OS networking. The default is max(384MB, executor_memory * 0.1). If your job uses pandas UDFs, Arrow-based operations, or reads ORC/Parquet with native readers, increase overhead to at least 20% of executor memory. Skimping here causes container killed by YARN — a silent OOM that looks like an executor lost.

3. Fixing OutOfMemoryError

Spark OOM errors come in two distinct flavours. Mixing up the fix is the most common tuning mistake.

Java heap OOM (GC overhead exceeded)

java.lang.OutOfMemoryError: GC overhead limit exceeded — the JVM is spending more than 98% of time in garbage collection. Causes: too many cached DataFrames, wide schemas with hundreds of columns, or excessive object creation in UDFs.

  • Increase spark.executor.memory.
  • Unpersist DataFrames after use: df.unpersist().
  • Use spark.memory.fraction (default 0.6) to give more heap to execution vs storage. Raise to 0.7–0.75 if you have few cached DataFrames but large aggregations.
  • Replace Python UDFs with native Spark SQL functions — Python UDFs serialise each row across the JVM/Python boundary, creating enormous GC pressure.

Container killed by YARN / OOM Killer

Container killed by YARN for exceeding memory limits — the OS OOM killer terminated the JVM process because total container memory (JVM heap + overhead) exceeded the YARN container allocation.

  • Increase spark.executor.memoryOverhead — this is almost always the fix for YARN kills.
  • If using pandas UDFs or Arrow: add spark.sql.execution.arrow.pyspark.enabled true and ensure overhead is at least 25% of executor memory.
  • Check for memory leaks in broadcast variables — very large broadcasts that are not unpersisted accumulate off-heap.
# Diagnose OOM type: check YARN container memory limit vs actual usage
# In EMR: yarn logs -applicationId application_XXX | grep -i "memory\|killed\|oom"
# In Databricks: cluster event log → "executorRemoved" with reason

# Fix for YARN container kill (most common case):
spark.executor.memoryOverhead  6g   # was default ~3.7g, now explicit
spark.memory.offHeap.enabled   true
spark.memory.offHeap.size      4g   # native memory for column vectors

4. Eliminating Data Skew

Data skew is the single most common cause of slow Spark jobs. It happens when data is unevenly distributed across partitions — one task processes 10 GB while 199 tasks process 50 MB each. The job is only as fast as the slowest task.

The canonical example: joining a user event table on user_id where 5% of users generate 60% of events (power users, bots, internal accounts). Those users land in a handful of partitions that take 10× longer than the rest.

Technique 1: Salting for GroupBy / Aggregation skew

from pyspark.sql import functions as F

SALT_BUCKETS = 20

# Phase 1: salt the key and aggregate within salted partitions
df_salted = df.withColumn(
    "salt", (F.rand() * SALT_BUCKETS).cast("int")
)

df_partial = (
    df_salted
    .groupBy("user_id", "salt")
    .agg(F.sum("amount").alias("partial_sum"), F.count("*").alias("partial_count"))
)

# Phase 2: merge partials — now each partition is evenly sized
df_result = (
    df_partial
    .groupBy("user_id")
    .agg(F.sum("partial_sum").alias("total_amount"), F.sum("partial_count").alias("event_count"))
)

Technique 2: Skewed join with AQE (Spark 3.0+)

# Enable AQE skew join handling — auto-detects and splits skewed partitions
spark.conf.set("spark.sql.adaptive.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")
# Partition is skewed if it's > this threshold AND > 5× median partition size
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes", "256m")
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionFactor", "5")

# With AQE enabled, Spark splits skewed partitions automatically at runtime.
# No code changes needed — just enable and let the optimizer handle it.
df_joined = events.join(users, "user_id", "left")
When AQE skew join isn't enough: AQE works well for moderate skew (1 partition = 5–10× median). For extreme skew — a single null key that represents 30% of the dataset — filter it out first, process it separately, and union the results. AQE cannot split a partition that contains only one logical key value.

5. Shuffle Tuning

The shuffle is the most expensive operation in Spark — it involves serialising data, writing to disk, transferring over the network, and deserialising on the target executor. Minimising shuffles and sizing them correctly is the highest-leverage optimisation available.

Partition count

spark.sql.shuffle.partitions defaults to 200 — fine for small datasets, far too low for anything above 10 GB, and wildly too high for tiny datasets (creating thousands of 1 KB files on S3). The target partition size is 100–200 MB of uncompressed data. 

# Rule of thumb: shuffle_partitions = total_shuffle_data_GB * 5
# For a 50 GB shuffle → 250 partitions. For a 500 GB shuffle → 2500 partitions.

# With AQE enabled, set this to a high upper bound — AQE coalesces automatically:
spark.conf.set("spark.sql.shuffle.partitions", "2000")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
# AQE coalesces down to this minimum (don't go below 1 per core):
spark.conf.set("spark.sql.adaptive.coalescePartitions.minPartitionNum", "50")

Avoiding unnecessary shuffles

  • Pre-partition your writes by the join key so downstream jobs can skip the shuffle stage entirely: df.repartition("user_id").write.partitionBy("date").
  • Use coalesce() over repartition() when reducing partition count — coalesce avoids a full shuffle by combining local partitions.
  • Cache strategically — if a DataFrame is used in multiple downstream operations, df.cache() or df.persist(StorageLevel.MEMORY_AND_DISK) eliminates recomputation from the lineage.

6. Adaptive Query Execution (AQE)

AQE (introduced in Spark 3.0) re-optimises the query plan at runtime, using statistics collected from completed shuffle stages. It is one of the most impactful Spark features of the last five years. Enable it on every production job running Spark 3.x.

AQE FeatureWhat it doesConfig flag
Partition coalescing Merges small post-shuffle partitions into larger ones, eliminating thousands of tiny tasks coalescePartitions.enabled
Skew join handling Splits skewed partitions and replicates the matching side — no code changes needed skewJoin.enabled
Dynamic join switching Converts a planned sort-merge join to a broadcast join if one side is small enough after the shuffle localShuffleReader.enabled 
# Full AQE configuration block — safe to apply to all Spark 3.x jobs
spark.conf.set("spark.sql.adaptive.enabled",                              "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled",            "true")
spark.conf.set("spark.sql.adaptive.coalescePartitions.minPartitionNum",     "50")
spark.conf.set("spark.sql.adaptive.advisoryPartitionSizeInBytes",           "128m")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled",                     "true")
spark.conf.set("spark.sql.adaptive.skewJoin.skewedPartitionThresholdInBytes", "256m")
spark.conf.set("spark.sql.adaptive.localShuffleReader.enabled",             "true")

7. Join Strategies: Broadcast, Sort-Merge & Bucket Joins

Broadcast join

When one side of the join fits in memory (<10 MB by default, configurable), Spark broadcasts it to all executors — no shuffle needed. This is the fastest join. Increase the auto-broadcast threshold for large lookup tables that fit in executor memory: 

# Increase from default 10 MB to 100 MB — safe for most dim tables
spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "104857600")  # 100 MB

# Or force a broadcast hint on a specific DataFrame:
from pyspark.sql.functions import broadcast
df_result = large_df.join(broadcast(small_dim), "dim_id")
Broadcast OOM: Each executor loads the entire broadcast table into memory. A 500 MB broadcast on a 50-executor cluster uses 25 GB of total executor heap — across all nodes simultaneously. If you see executor OOM immediately after a join, check whether a hint is forcing a broadcast of a table too large for your executor memory budget.

Bucket join (the shuffle eliminator for repeated joins)

If you join the same two large tables repeatedly across multiple jobs, bucket them once and eliminate the shuffle permanently. Bucketing pre-partitions and sorts data by the join key at write time: 

# Write events bucketed by user_id — do this once during pipeline setup
events.write.bucketBy(256, "user_id").sortBy("user_id").saveAsTable("events_bucketed")
users.write.bucketBy(256, "user_id").sortBy("user_id").saveAsTable("users_bucketed")

# Subsequent jobs can join with zero shuffle — Spark detects compatible bucketing
spark.conf.set("spark.sql.sources.bucketing.enabled", "true")
df = spark.table("events_bucketed").join(spark.table("users_bucketed"), "user_id")
# Check explain() — you should see "BucketedScan" and no "Exchange" nodes

8. File Format & Partition Optimisation

Parquet over CSV, always

Parquet's columnar storage and built-in statistics (min/max per row group) allow Spark to skip entire chunks of data without reading them — called predicate pushdown. A filtered read on a 1 TB Parquet table can read less than 10 GB if the filter aligns with column statistics. CSV has none of this.

Partition pruning

Physical partition layout on S3/HDFS is the second tier of pushdown. Always partition your data by the columns most frequently used in WHERE clauses — typically date, region, or event_type. A job reading WHERE date = '2026-06-05' on a date-partitioned table reads only one day's data regardless of total table size.

Small file problem

Too many small Parquet files (under 32 MB) cause excessive S3 LIST operations and per-file JVM overhead. After a high-concurrency write or after many micro-batch appends, compact files: 

# Compact small files in a partition — read and rewrite with target file size
spark.conf.set("spark.sql.files.maxPartitionBytes", "134217728")  # 128 MB per task

df = spark.read.parquet("s3://bucket/silver/events/date=2026-06-05/")
# repartition to target ~200 MB files
target_files = max(1, df.rdd.getNumPartitions() // 4)
df.coalesce(target_files).write.mode("overwrite").parquet("s3://bucket/silver/events/date=2026-06-05/")

9. Production Tuning Checklist

Before deploying any Spark job to production, run through this checklist:

Real numbers from production: On the Banco Itaú data platform at NTT DATA, applying these techniques to a 2-hour daily Spark job — enabling AQE, fixing a null-key skew with filter+union, and increasing memoryOverhead — reduced runtime to 47 minutes and eliminated all executor OOM failures. The same job had been failing intermittently for three months before the root causes were identified with the Spark UI.

Spark tuning is a diagnostic discipline, not a configuration game. Measure first, change one thing at a time, and validate with the Spark UI after every change. The payoff is real — well-tuned Spark jobs consistently run 40–70% faster and cost a fraction of their unoptimised baseline.
