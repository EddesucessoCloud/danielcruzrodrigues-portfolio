Redshift Serverless vs Provisioned: The Real Cost Model at Scale

The $14,000 surprise bill

A team I know migrated from a ra3.4xlarge provisioned cluster to Redshift Serverless expecting to save money. Their workload was bursty — analytics dashboards refreshed every 30 minutes throughout business hours. Three months later, they were paying $14,000/month instead of $5,400. Redshift Serverless is not cheaper by default. It is cheaper for the right query pattern. This post shows you how to determine which pattern you have before you migrate.

How Redshift Serverless billing works

Redshift Serverless charges in RPU-seconds (Redshift Processing Units). Each RPU costs $0.375/hour ($0.0001041/second). You configure a base RPU (minimum 8 RPUs) and a max RPU (up to 512). The namespace scales up to max RPU when queries arrive and scales down after 6 minutes of inactivity.

The billing trap: you pay for the full RPU capacity for the duration of your query, even if the query only uses 10% of it. A 5-second query at 128 RPU costs 128 × 5 × $0.0001041 = $0.067. Run 10,000 such queries per day and that's $670/day — $20,100/month. The same ra3.4xlarge provisioned cluster costs $5,400/month flat.

ℹ RPU billing formula

Monthly cost = (average active RPUs) × (seconds active per day) × 30 × $0.0001041. The break-even point vs provisioned depends entirely on your duty cycle — what percentage of the day your workload is actively running queries.

The duty cycle decision

This is the only metric that matters for the Serverless vs Provisioned decision: what fraction of each day is your cluster actively executing queries? A cluster that runs ad-hoc queries from 9AM–6PM (9 hours of 24 = 37.5% duty cycle) is very different from a batch pipeline that runs 24×7.

Calculate your current duty cycle

-- Run on your existing provisioned cluster
-- Shows query activity distribution by hour over the last 30 days
SELECT
  DATE_TRUNC('hour', starttime)          AS query_hour,
  COUNT(*)                               AS query_count,
  SUM(datediff('second', starttime, endtime)) AS total_query_seconds,
  AVG(datediff('second', starttime, endtime)) AS avg_duration_sec
FROM stl_query
WHERE starttime >= DATEADD('day', -30, GETDATE())
  AND userid > 1  -- exclude system users
GROUP BY 1
ORDER BY 1 DESC;

Concurrency: where Serverless breaks down

Redshift Serverless does not automatically increase concurrency scaling the way provisioned clusters do with Concurrency Scaling. When 50 dashboard queries land simultaneously on a Serverless namespace configured at 32 RPU, those queries queue. The queue drains sequentially. At 128 RPU max, you get roughly 8–12 concurrent medium queries. That's it.

Provisioned clusters with Concurrency Scaling add temporary clusters automatically when the main cluster queue exceeds a threshold. Concurrency Scaling costs $0.024/cluster-minute, but AWS gives you 1 free Concurrency Scaling cluster-hour per day per cluster — enough to absorb most burst scenarios in a standard 8-hour business day.

⚠ Watch out

Redshift Serverless query queue depth is not exposed in CloudWatch by default. If your queries are silently queuing, you will see high QueryDuration but not the cause. Enable enhanced monitoring and check QueriesCompletedPerSecond vs QueriesRunning to detect queuing pressure.

The hidden cost: Managed Storage

Both Serverless and Provisioned ra3 nodes charge for Redshift Managed Storage at $0.024/GB-month. There is no difference between the two deployment modes here. Where teams get surprised is the minimum: Serverless has a minimum storage charge equivalent to what the base RPU configuration would need to hold your dataset. At 8 base RPUs with 2TB of data, storage costs $48/month regardless of query activity.

When to use Serverless without hesitation

Use Redshift Serverless when: (1) you have a dev or staging environment where the cluster is idle >80% of the time; (2) your analytics workload is genuinely ad-hoc with no predictable pattern; (3) you need Redshift without the operational overhead of cluster management and are willing to pay the premium. Use provisioned when you have any SLA on query latency, high concurrency requirements, or continuous ETL pipelines.

✓ Tip

For mixed workloads, use provisioned for your production ETL pipeline and Serverless for the ad-hoc analytics namespace. You can share data via Redshift Data Sharing with zero data copying — the ETL writes to the provisioned cluster, Serverless reads from it in real time.

Redshift Data Sharing setup

-- On provisioned cluster (producer): create datashare
CREATE DATASHARE analytics_share;
ALTER DATASHARE analytics_share ADD SCHEMA public;
ALTER DATASHARE analytics_share ADD ALL TABLES IN SCHEMA public;

-- Grant access to Serverless namespace consumer
GRANT USAGE ON DATASHARE analytics_share
TO NAMESPACE 'xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx'; -- Serverless namespace ID

-- On Serverless namespace (consumer): create database from datashare
CREATE DATABASE prod_data FROM DATASHARE analytics_share
OF NAMESPACE 'yyyyyyyy-yyyy-yyyy-yyyy-yyyyyyyyyyyy'; -- provisioned namespace ID
