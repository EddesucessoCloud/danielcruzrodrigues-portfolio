Data Quality with Great Expectations on AWS Glue: A Production Setup

The Glue job that passed but delivered garbage for 11 days

A Glue ETL job was processing daily sales data from an RDS source. The job completed without errors every morning. What nobody noticed: the source team had changed a column name from sale_amount to amount_usd twelve days ago. The Glue job read the missing column as null, summed nulls as zero, and wrote zeros to the gold layer. Eleven days of zero-revenue records landed in Redshift. The finance dashboard looked correct — all green — because zero is a valid number. Great Expectations with a expect_column_values_to_not_be_null check on sale_amount would have caught this on day one.

Great Expectations on Glue: the packaging challenge

Great Expectations (GX) is a Python library. Running it inside a Glue job requires packaging it as a Python wheel and passing it via the Glue job's --additional-python-modules parameter. GX 0.18+ works with Glue 4.0 (Python 3.10, Spark 3.3). Earlier Glue versions require pinning GX to 0.15.x due to dependency conflicts with the pre-installed boto3 version.

# Package GX for Glue
pip install great-expectations==0.18.15 --target ./gx_dist
cd gx_dist && zip -r ../great_expectations.zip . && cd ..
aws s3 cp great_expectations.zip s3://your-glue-assets/libs/

# In Terraform or Glue job definition:
# --additional-python-modules great_expectations==0.18.15
# Or use wheel file: upload to S3 and reference via --extra-py-files
⚠ Watch out

Great Expectations has significant transitive dependencies (ruamel.yaml, mistune, jinja2, marshmallow). When using --additional-python-modules, version conflicts with Glue's pre-installed packages are common. Pin versions explicitly and test in a Glue development endpoint before deploying. GX startup adds 30–45 seconds to Glue job cold start time.

Setting up the context and expectation suite

import great_expectations as gx
from great_expectations.core.batch import RuntimeBatchRequest
from pyspark.sql import SparkSession

# Use ephemeral (in-memory) Data Context — no filesystem required in Glue
context = gx.get_context(mode="ephemeral")

# Configure S3 as Data Docs site (publishes HTML quality report to S3)
context.add_store(
    store_name="validation_result_store",
    store_config={
        "class_name": "ValidationsStore",
        "store_backend": {
            "class_name": "TupleS3StoreBackend",
            "bucket": "your-datalake",
            "prefix": "great_expectations/validations/"
        }
    }
)

context.add_data_docs_site(
    site_name="s3_site",
    site_config={
        "class_name": "SiteBuilder",
        "store_backend": {
            "class_name": "TupleS3StoreBackend",
            "bucket": "your-datalake",
            "prefix": "great_expectations/data_docs/"
        },
        "site_index_builder": {"class_name": "DefaultSiteIndexBuilder"}
    }
)

Defining expectations that catch real problems

from great_expectations.core import ExpectationSuite, ExpectationConfiguration

suite = ExpectationSuite(expectation_suite_name="sales_daily_v1")

# Structural checks — catch schema drift immediately
suite.add_expectation(ExpectationConfiguration(
    expectation_type="expect_table_columns_to_match_ordered_list",
    kwargs={"column_list": ["order_id", "customer_id", "sale_amount", "sale_date", "region"]}
))

# Completeness — no nulls on critical columns
suite.add_expectation(ExpectationConfiguration(
    expectation_type="expect_column_values_to_not_be_null",
    kwargs={"column": "sale_amount", "mostly": 1.0}
))

# Range checks — catch unit errors (cents vs dollars, negative values)
suite.add_expectation(ExpectationConfiguration(
    expectation_type="expect_column_values_to_be_between",
    kwargs={"column": "sale_amount", "min_value": 0.01, "max_value": 1000000}
))

# Volume check — catch data pipeline failures that produce empty datasets
suite.add_expectation(ExpectationConfiguration(
    expectation_type="expect_table_row_count_to_be_between",
    kwargs={"min_value": 1000, "max_value": 10000000}
))

# Referential integrity — region must be a known value
suite.add_expectation(ExpectationConfiguration(
    expectation_type="expect_column_values_to_be_in_set",
    kwargs={"column": "region", "value_set": ["US", "EU", "APAC", "LATAM"]}
))

Running validations inside the Glue job

from great_expectations.datasource.fluent import SparkDatasource

# Register the Spark DataFrame as a GX data source
datasource = context.sources.add_spark(name="glue_spark")
data_asset = datasource.add_dataframe_asset(name="sales_data")

# Validate — this triggers all expectations in the suite
batch_request = data_asset.build_batch_request(dataframe=sales_df)
checkpoint = context.add_or_update_checkpoint(
    name="sales_checkpoint",
    validations=[{
        "batch_request": batch_request,
        "expectation_suite_name": "sales_daily_v1"
    }]
)

results = checkpoint.run()
context.build_data_docs()  # Publish HTML report to S3

# Fail the Glue job if critical expectations fail
if not results.success:
    failed = [
        f"{v['expectation_config']['expectation_type']} on {v['expectation_config']['kwargs'].get('column', 'table')}"
        for r in results.run_results.values()
        for v in r['results'] if not v['success']
    ]
    raise ValueError(f"Data quality failed: {', '.join(failed[:5])}")

Alerting on DQ failures: SNS + CloudWatch

import boto3
import json

sns = boto3.client('sns', region_name='us-east-1')

def publish_dq_alert(suite_name: str, failed_expectations: list, run_date: str):
    message = {
        "alert_type": "DATA_QUALITY_FAILURE",
        "pipeline": suite_name,
        "run_date": run_date,
        "failed_count": len(failed_expectations),
        "failures": failed_expectations[:10],  # cap at 10 for readability
        "data_docs_url": f"https://your-datalake.s3.amazonaws.com/great_expectations/data_docs/index.html"
    }
    sns.publish(
        TopicArn="arn:aws:sns:us-east-1:123456789012:data-quality-alerts",
        Subject=f"[DQ FAILURE] {suite_name} — {len(failed_expectations)} checks failed",
        Message=json.dumps(message, indent=2)
    )
✓ Tip

Separate expectations into "blocking" (job fails, data not written) and "warning" (data written, alert sent). A missing column is blocking. A slightly higher-than-usual null rate in an optional field is a warning. Treating all DQ failures as blocking causes pipeline outages over minor data drift; treating all as warnings defeats the purpose.

Performance: GX overhead at scale

Great Expectations runs on the Spark driver for dataset-level expectations (row count, column existence) and triggers Spark actions for column-level checks (null counts, value ranges). At 100M rows, a suite with 20 expectations adds approximately 90 seconds to job runtime — each expectation triggers a separate Spark job. Use mostly thresholds (e.g., "mostly": 0.99) to allow GX to short-circuit after sampling 1% of the dataset for column-level checks, reducing overhead to 15–20 seconds on large datasets.
