1. What is the Medallion Architecture?

The medallion architecture (also called the multi-hop or lakehouse architecture) organises a data lake into three progressively refined layers — Bronze, Silver, and Gold. Each layer has a clear contract: what data it contains, what quality it guarantees, and who consumes it.

Originally popularised by Databricks for Delta Lake on Azure, it maps equally well to an AWS stack: S3 for storage, Delta Lake or Apache Iceberg for table format, AWS Glue or EMR for compute, and the Glue Data Catalog as the central metastore. I implemented this architecture at NTT DATA for the Banco Itaú data platform, processing 20M+ financial events daily through three layers before delivery to Redshift and downstream BI tools.

┌─────────────────────────────────────────────────────────┐
│ BRONZE · Raw, immutable, append-only │
│ Sources: Kinesis, S3 uploads, JDBC, APIs │
│ Format: JSON / CSV / Avro as-is from source │
│ Latency: minutes (batch) or seconds (streaming) │
└──────────────────────────┬──────────────────────────────┘
│ Glue / EMR Spark job
▼
┌─────────────────────────────────────────────────────────┐
│ SILVER · Cleansed, schema-enforced, deduplicated │
│ Consumers: Data Science, exploratory SQL (Athena) │
│ Format: Parquet / Delta Lake, partitioned by date │
│ Latency: hours (batch SLA) │
└──────────────────────────┬──────────────────────────────┘
│ Glue / EMR aggregation job
▼
┌─────────────────────────────────────────────────────────┐
│ GOLD · Business aggregates, serving layer │
│ Consumers: BI dashboards, Redshift, ML features │
│ Format: Delta Lake / Iceberg, denormalised wide tables │
│ Latency: daily (business SLA) │
└─────────────────────────────────────────────────────────┘

2. The AWS Stack for Each Layer

LayerComputeStorageCatalogQuery

Silver AWS Glue Spark, EMR on EC2 S3 + Delta Lake Glue Catalog (conformed tables) Athena, Redshift Spectrum

3. S3 Bucket Layout & Naming Conventions

A consistent S3 layout is the foundation of the entire architecture. The prefixes dictate IAM policy boundaries, lifecycle rules, and partition pruning efficiency. Here is the structure I use in production: 

# Recommended S3 structure for a medallion data lake
s3://<org>-datalake-<env>/

# ── Bronze (raw, immutable) ───────────────────────────────
  bronze/
    <source-system>/
      <entity>/
        year=YYYY/month=MM/day=DD/hour=HH/
          <uuid>.json.gz

# ── Silver (cleansed, schema-enforced) ───────────────────
  silver/
    <domain>/
      <entity>/
        date=YYYY-MM-DD/
          part-00000-<uuid>.parquet

# ── Gold (business aggregates) ───────────────────────────
  gold/
    <domain>/
      <aggregate_name>/
        date=YYYY-MM-DD/
          part-00000-<uuid>.parquet

# ── System prefixes ──────────────────────────────────────
  _delta_log/         # Delta Lake transaction log (per table root)
  _spark_metadata/    # Spark streaming checkpoint
  _tmp/               # Staging, cleaned up post-job
  _quarantine/        # Failed quality checks, never deleted
Single bucket vs multi-bucket: Use a single bucket per environment (dev/staging/prod) with prefix-based IAM and Lake Formation policies. Multi-bucket setups (one per layer) feel cleaner but create cross-bucket replication complexity and inflate S3 request costs on LIST operations. The prefix approach gives the same isolation with simpler IAM.

4. Bronze Layer: Raw Ingestion

The bronze layer has one job: land data from source systems as fast as possible, exactly as it arrived, without any transformation. This is your audit log — every schema change, every corrupt record, every API quirk is preserved here forever.

Ingestion pattern: Kinesis → Firehose → S3

# Kinesis Firehose configuration (Terraform excerpt)
# Delivers raw JSON events to Bronze with Hive-compatible partitioning
resource "aws_kinesis_firehose_delivery_stream" "bronze_events" {
  name        = "bronze-events-stream"
  destination = "extended_s3"

  extended_s3_configuration {
    role_arn           = aws_iam_role.firehose_role.arn
    bucket_arn         = aws_s3_bucket.datalake.arn
    prefix             = "bronze/events/year=!{timestamp:yyyy}/month=!{timestamp:MM}/day=!{timestamp:dd}/hour=!{timestamp:HH}/"
    error_output_prefix = "_quarantine/firehose-errors/!{firehose:error-output-type}/"
    buffering_size     = 128   # MB — flush when buffer hits 128 MB
    buffering_interval = 300   # seconds — flush every 5 min at minimum
    compression_format = "GZIP"
  }
}
Never transform in Bronze: Resist the temptation to apply any business logic, type casting, or filtering during ingestion. If your transformation logic turns out to be wrong (and it will), you need the raw data to reprocess. Bronze is your time machine — it should always let you replay the entire history.

5. Silver Layer: Cleansed & Conformed

The silver layer applies schema enforcement, deduplication, null handling, and type casting. The output is trusted by data scientists and analysts for exploratory work — they should never need to handle malformed data. 

# bronze_to_silver.py — core transformation pattern
from pyspark.sql import SparkSession, functions as F
from pyspark.sql.types import StructType, StructField, StringType, LongType, TimestampType
from delta.tables import DeltaTable

BRONZE_PATH = "s3://org-datalake-prod/bronze/events/"
SILVER_PATH = "s3://org-datalake-prod/silver/events/"
RUN_DATE    = "2026-06-05"

# Enforced schema — bronze may have extra fields; we only promote what we own
SILVER_SCHEMA = StructType([
    StructField("event_id",   StringType(),    nullable=False),
    StructField("user_id",    StringType(),    nullable=False),
    StructField("event_type", StringType(),    nullable=True),
    StructField("amount_brl", LongType(),      nullable=True),
    StructField("event_ts",   TimestampType(), nullable=False),
])

df_bronze = (
    spark.read
    .schema(SILVER_SCHEMA)
    .json(f"{BRONZE_PATH}year={RUN_DATE[:4]}/month={RUN_DATE[5:7]}/day={RUN_DATE[8:10]}/")
)

df_silver = (
    df_bronze
    # Drop rows missing primary key — send to quarantine separately
    .filter(F.col("event_id").isNotNull() & F.col("user_id").isNotNull())
    # Dedup: keep latest record per event_id in this batch
    .dropDuplicates(["event_id"])
    # Normalise
    .withColumn("event_type", F.lower(F.trim(F.col("event_type"))))
    .withColumn("date", F.to_date(F.col("event_ts")))
    .withColumn("_ingested_at", F.current_timestamp())
)

# Upsert into Delta Silver table (idempotent re-runs)
if DeltaTable.isDeltaTable(spark, SILVER_PATH):
    DeltaTable.forPath(spark, SILVER_PATH)\
        .alias("t")\
        .merge(df_silver.alias("s"), "t.event_id = s.event_id")\
        .whenMatchedUpdateAll()\
        .whenNotMatchedInsertAll()\
        .execute()
else:
    df_silver.write.format("delta").partitionBy("date").save(SILVER_PATH)

Why Delta Lake for Silver and Gold?

Plain Parquet on S3 is write-once. To update or delete records (GDPR erasure, late-arriving corrections), you have to rewrite entire partitions. Delta Lake adds ACID transactions, a transaction log, and MERGE/UPDATE/DELETE semantics on top of Parquet — without moving away from S3. On AWS, use the open-source delta-spark library with EMR or Glue.

6. Gold Layer: Business Aggregates

Gold tables are pre-aggregated, denormalised, and optimised for read performance — not storage efficiency. A BI analyst or data scientist consuming gold data should never need to join more than 2 tables or filter by anything other than a date range. 

# silver_to_gold.py — daily user transaction summary
df_gold = (
    spark.read.format("delta").load(SILVER_PATH)
    .filter(F.col("date") == RUN_DATE)
    .groupBy("user_id", "event_type", "date")
    .agg(
        F.count("*").alias("event_count"),
        F.sum("amount_brl").alias("total_amount_brl"),
        F.countDistinct("event_id").alias("unique_events"),
        F.max("event_ts").alias("last_event_ts"),
    )
    .withColumn("_computed_at", F.current_timestamp())
)

# Z-order on frequently filtered columns for faster Athena / Redshift Spectrum reads
# (Delta Lake specific — rewrites files sorted by these columns)
DeltaTable.forPath(spark, GOLD_PATH)\
    .optimize().executeZOrderBy("user_id", "event_type")

7. Data Quality Gates

Every layer transition must be gated by a quality check. Data that fails a quality rule should never advance to the next layer — it should land in quarantine and trigger an alert.

Quality rules per layer

Layer transitionRule typeExample checkOn failure
Bronze → Silver Schema completeness event_id IS NOT NULL for >99% of rows Quarantine bad rows, proceed with clean rows
Bronze → Silver Referential integrity event_type IN known set of values Flag unknowns, alert, do not block
Silver → Gold Aggregate freshness Row count within ±10% of 7-day average Halt pipeline, page on-call
Silver → Gold Business constraint amount_brl >= 0 for all transactions Quarantine, alert, halt gold write 
# Lightweight quality gate using native PySpark — no extra libraries needed
def assert_quality(df, col, condition, threshold=0.99, name=""):
    total   = df.count()
    passing = df.filter(condition).count()
    ratio   = passing / total if total > 0 else 0
    if ratio < threshold:
        raise ValueError(ff"Quality gate FAILED [{name}]: {ratio:.2%} < {threshold:.2%}")
    print(ff"Quality gate PASSED [{name}]: {ratio:.2%}")

assert_quality(df_silver, "event_id", F.col("event_id").isNotNull(), threshold=0.999, name="event_id not null")
assert_quality(df_silver, "amount_brl", F.col("amount_brl") >= 0, threshold=1.0, name="non-negative amount")

8. Governance with Lake Formation

As the number of consumers grows, IAM bucket policies become unmanageable. AWS Lake Formation provides a centralised permission model over the Glue Data Catalog — column-level, row-level, and tag-based access control without changing S3 bucket policies.

  • Data Scientists get SELECT on silver.* — all tables, all columns.
  • BI Analysts get SELECT on gold.* — pre-aggregated, no PII columns.
  • ML Pipelines get SELECT on specific gold.* feature tables only.
  • ETL roles (Glue, EMR) get ALL on their own layer prefix only — a Bronze ETL role cannot write to Silver.
Lake Formation gotcha: Lake Formation permissions layer on top of IAM, not replace it. A principal needs both a valid IAM role AND a Lake Formation grant to access a table. Debugging access denials requires checking both layers — the Lake Formation console shows "Data lake access denied" separately from IAM policy denials. Enable CloudTrail data events on the S3 bucket to audit all access at the object level.

9. Orchestration with Step Functions & EventBridge

The pipeline between layers needs reliable orchestration — not a cron job. AWS Step Functions with EventBridge Scheduler gives you visual state machines, automatic retries, failure notifications, and execution history. 

# EventBridge Scheduler: trigger Bronze-to-Silver pipeline at 02:00 UTC daily
resource "aws_scheduler_schedule" "daily_etl" {
  name                         = "bronze-to-silver-daily"
  schedule_expression          = "cron(0 2 * * ? *)"
  schedule_expression_timezone = "UTC"

  flexible_time_window { mode = "OFF" }

  target {
    arn      = aws_sfn_state_machine.medallion_pipeline.arn
    role_arn = aws_iam_role.scheduler_role.arn
    input    = jsonencode({
      run_date    = "$.time"  # injected from EventBridge event
      environment = "prod"
    })
    retry_policy {
      maximum_retry_attempts       = 2
      maximum_event_age_in_seconds = 3600
    }
  }
}
Step Functions Express vs Standard: Use Standard workflows for the medallion pipeline — they persist execution history for 90 days, support exactly-once execution semantics, and can run up to 1 year. Express workflows are cheaper but limited to 5-minute maximum duration and at-least-once semantics, making them unsuitable for multi-hour ETL jobs.

10. Common Pitfalls & How to Avoid Them

PitfallSymptomFix
Transforming in Bronze Cannot replay history when logic changes Land raw bytes, transform only in Silver
Over-partitioning Silver Thousands of 1 KB Parquet files, slow S3 LIST Partition by date only; use Z-order for secondary dimensions
No idempotent writes Duplicate data on retry after failure Use Delta MERGE with primary key; Glue job bookmarks for Bronze
Gold tables too normalised BI tools joining 5 tables, slow dashboards Denormalise into wide tables; pre-join dimensions at write time
No quarantine layer Bad records silently dropped; no audit trail Write every rejected record to _quarantine/ with error reason
Skipping data quality gates Corrupt gold data surfaced in BI dashboards Assert row counts and key constraints at every layer boundary
Single S3 bucket for all envs Dev jobs writing to prod paths Separate buckets per environment, enforce with IAM conditions
From production at NTT DATA / Banco Itaú: The pitfall that caused the most incidents was skipping idempotent writes in the early Silver jobs. A transient Glue job failure on retry would duplicate the day's events because we were using .mode("append") instead of Delta MERGE. After switching to MERGE on event_id, reruns became completely safe and the deduplication issue disappeared — removing an entire class of data incidents.

The medallion architecture is not complex — it is disciplined. Bronze for raw immutability, Silver for trusted data, Gold for business-ready aggregates. Three layers, clear contracts between them, quality gates enforced at each boundary. That discipline is what makes a data lake usable at scale instead of a swamp.
