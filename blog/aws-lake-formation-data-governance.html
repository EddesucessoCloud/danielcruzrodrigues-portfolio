AWS Lake Formation: The Data Governance Layer Most Teams Skip

The compliance audit that exposed 47 over-privileged roles

An audit of a production data lake found that 47 IAM roles had s3:GetObject on the entire data lake bucket — including a role used by a vendor integration that should only have accessed aggregate reports. Every engineer with a Glue or Athena role could read the raw PII tables. Nobody had designed it this way — it had just accumulated over 18 months of "we'll fix the permissions later." Lake Formation was the fix. It took 3 days to implement what IAM alone cannot provide: table-level and column-level access control without touching S3 bucket policies.

What Lake Formation actually controls

AWS Lake Formation sits on top of the Glue Data Catalog and S3. It adds a permissions model that operates at the database, table, and column level — separate from and additive to IAM. When Lake Formation is enabled, access to Glue Catalog resources requires both IAM permissions (to call Glue/Athena APIs) AND Lake Formation grants. IAM alone is insufficient. This is the "hybrid access mode" that most documentation underexplains.

ℹ Permission model

Lake Formation uses a "least privilege intersection" model: a principal needs both IAM permission to call the service API AND a Lake Formation grant on the specific resource. Removing the Lake Formation grant revokes access regardless of what IAM says. This is why Lake Formation is powerful — it is an additional control plane, not a replacement.

Enable Lake Formation and register your S3 location

resource "aws_lakeformation_resource" "datalake_bucket" {
  arn = "arn:aws:s3:::your-datalake-bucket"
  # Lake Formation manages access to this S3 path
  # Glue/Athena will use LF-registered credentials, not role credentials
}

resource "aws_lakeformation_permissions" "data_engineer_table" {
  principal   = "arn:aws:iam::123456789012:role/DataEngineerRole"
  permissions = ["SELECT", "DESCRIBE"]

  table {
    database_name = "events"
    name          = "user_events"
  }
}

resource "aws_lakeformation_permissions" "analyst_columns" {
  principal   = "arn:aws:iam::123456789012:role/AnalystRole"
  permissions = ["SELECT"]

  table_with_columns {
    database_name = "events"
    name          = "user_events"
    column_names  = ["event_type", "event_ts", "session_id"]
    # user_id, email, phone are excluded — analyst cannot query PII columns
  }
}

Column-level security: the feature IAM cannot replicate

IAM bucket policies operate on S3 object paths — you cannot restrict access to a column within a Parquet file via IAM. Lake Formation's column-level security intercepts queries at the Glue Catalog level: when Athena or Redshift Spectrum reads a table, Lake Formation verifies the column-level grants and rewrites the query plan to exclude unauthorized columns. The calling principal never receives the restricted data — it is not returned from S3 reads at all.

⚠ Watch out

Column-level security applies to Athena, Redshift Spectrum, and Glue ETL jobs when using the Glue Catalog. It does NOT apply to direct S3 access via Spark or EMR if the job bypasses the Glue Catalog. A data engineer with s3:GetObject IAM permissions can still read all columns from the Parquet file directly. Lake Formation only controls catalog-mediated access.

Cross-account data sharing with Lake Formation

Lake Formation cross-account grants let you share Glue Catalog tables with principals in different AWS accounts without copying data. The consumer account queries the producer's S3 data directly, with Lake Formation enforcing the column-level grants across the account boundary. This is the technical foundation of a data mesh on AWS — each domain account owns its data, Lake Formation controls who can read what.

import boto3

lf = boto3.client('lakeformation', region_name='us-east-1')

# Producer account: grant cross-account access
lf.grant_permissions(
    Principal={'DataLakePrincipalIdentifier': '987654321098'},  # consumer account ID
    Resource={
        'TableWithColumns': {
            'CatalogId': '123456789012',  # producer catalog
            'DatabaseName': 'events',
            'Name': 'user_events',
            'ColumnNames': ['event_type', 'event_ts', 'region']
            # PII columns not included
        }
    },
    Permissions=['SELECT', 'DESCRIBE'],
    PermissionsWithGrantOption=[]  # consumer cannot re-grant
)

# Consumer account: accept the RAM resource share first, then query
# The consumer uses Athena pointed at the federated catalog
# No data copying — queries go to producer's S3 directly

Lake Formation vs Glue Catalog IAM: the migration trap

Before Lake Formation, Glue Catalog permissions were managed via IAM and S3 bucket policies. Enabling Lake Formation in "hybrid mode" — where both IAM and LF permissions control access — seems safe but creates a period where you have two permission systems in conflict. The migration approach that works: enable Lake Formation with IAM + Lake Formation access mode (hybrid), audit all existing IAM roles using the LF access control verifier, then revoke over-permissive S3 bucket policies once LF grants are confirmed.

✓ Migration sequence

1. Enable Lake Formation with hybrid mode (IAM + LF). 2. Register S3 locations. 3. Grant table-level permissions to existing roles. 4. Verify access with each principal. 5. Switch to LF-only mode. 6. Tighten S3 bucket policy to only allow Lake Formation service role. Do NOT skip step 4 — switching to LF-only mode with missing grants immediately breaks production jobs.

Data quality tags and automated governance

Lake Formation's LF-Tags let you attach arbitrary metadata to Catalog resources and grant permissions based on tag combinations rather than explicit table names. A tag sensitivity=pii applied to a table automatically restricts access for any role that lacks the matching LF-Tag permission. This scales governance: adding a new PII table automatically inherits the right access controls without a manual grant for each team.

# Create LF-Tag and apply to tables
lf.create_lf_tag(TagKey='sensitivity', TagValues=['pii', 'internal', 'public'])

lf.add_lf_tags_to_resource(
    Resource={'Table': {'DatabaseName': 'events', 'Name': 'user_events'}},
    LFTags=[{'TagKey': 'sensitivity', 'TagValues': ['pii']}]
)

# Grant access based on tag: any role with this grant can access ALL tables tagged sensitivity=public
lf.grant_permissions(
    Principal={'DataLakePrincipalIdentifier': 'arn:aws:iam::123456789012:role/PublicAnalystRole'},
    Resource={'LFTagPolicy': {
        'ResourceType': 'TABLE',
        'Expression': [{'TagKey': 'sensitivity', 'TagValues': ['public']}]
    }},
    Permissions=['SELECT', 'DESCRIBE']
)
