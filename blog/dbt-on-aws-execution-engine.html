dbt on AWS: Choosing the Right Execution Engine (Glue vs Redshift vs Athena)

The dbt-glue job that took 4 hours to do what dbt-redshift did in 12 minutes

A team migrated their dbt project from dbt-redshift to dbt-glue, expecting to save money and gain Spark's processing power. Their flagship model was a 60-table DAG producing a 500M-row fact table. On Redshift, the full run took 12 minutes. On Glue Spark with the same data, it took 4 hours — 5 minutes of actual computation and 3 hours 55 minutes of Glue job startup, Spark session initialization, and data format conversion overhead. They switched back. The lesson: dbt-glue is not "dbt but faster." It is a fundamentally different tool with different cost and performance characteristics.

The three adapters and what they actually do

dbt-redshift compiles SQL models into Redshift SQL and runs them via psycopg2. dbt-athena submits SQL to Athena Query Execution API and writes results to S3 as Parquet. dbt-glue converts SQL models to PySpark DataFrames and runs them inside a Glue Interactive Session or Glue Job. The execution model matters: Redshift and Athena are SQL push-down engines — your SQL runs natively. Glue converts it, introducing overhead and a different optimizer.

ℹ Startup cost reality

Athena: 2–5 second query startup. Redshift: 0 seconds (connected session, persistent cluster). Glue Interactive Session: 45–90 second warm-up per session. Glue Job: 3–8 minutes for Spark cluster initialization. For a dbt project with 80 models, Glue startup time alone adds 4–10 minutes to every run even if models themselves execute instantly.

dbt-athena: the best fit for a data lake

dbt-athena is the right choice when your data lives in S3 and you need dbt's modeling layer without a persistent compute cluster. Each model runs as an Athena query — you pay $5/TB scanned. For a project with 50 models each scanning 10GB (columnar, partitioned), a full run costs 50 × 0.01TB × $5 = $2.50. There is no cluster to provision, no idle cost, and models can be parallelized across Athena's default 20 concurrent query limit.

# profiles.yml — dbt-athena configuration
your_project:
  target: prod
  outputs:
    prod:
      type: athena
      s3_staging_dir: s3://your-datalake/athena-results/
      region_name: us-east-1
      database: dbt_prod
      schema: analytics
      work_group: dbt-workgroup          # Use a named workgroup for cost tracking
      num_retries: 3
      s3_data_dir: s3://your-datalake/dbt/
      s3_data_naming: uuid               # prevents table conflicts on concurrent runs
      lf_tags_database:                  # Optional Lake Formation tags
        team: analytics
        environment: prod
⚠ Watch out

dbt-athena materializations write intermediate results to S3. Incremental models on large tables can accumulate S3 objects rapidly — each run appends new Parquet files without compaction. After 30 days of daily incremental runs on a 1B-row table, you may have 720 small files. Add a periodic OPTIMIZE or compaction step (EMR or Athena CTAS) to maintain query performance.

dbt-redshift: the fastest SQL transformation on AWS

dbt-redshift is unambiguously the fastest option for SQL-based transformations when your data is in Redshift. Redshift's massively parallel query engine handles complex multi-join aggregations across 500M rows in 30–120 seconds. You get Redshift's distribution and sort key optimization, materialized views, and late-binding views for free. The limitation: your data must be in Redshift. Transformations that read from S3 (Spectrum) add latency and cost.

# profiles.yml — dbt-redshift
your_project:
  target: prod
  outputs:
    prod:
      type: redshift
      host: "{{ env_var('REDSHIFT_HOST') }}"
      port: 5439
      user: dbt_user
      password: "{{ env_var('REDSHIFT_PASSWORD') }}"
      dbname: analytics
      schema: dbt_prod
      threads: 8
      keepalives_idle: 240
      connect_timeout: 10
-- dbt model: optimized for Redshift distribution
-- models/fct_orders.sql
{{
  config(
    materialized='incremental',
    unique_key='order_id',
    dist='customer_id',        -- distribute on join key
    sort=['order_date', 'region'],  -- compound sort key for time-based filters
    incremental_strategy='delete+insert'
  )
}}

SELECT
  o.order_id,
  o.customer_id,
  o.order_date,
  o.region,
  SUM(li.quantity * li.unit_price) AS order_total
FROM {{ ref('stg_orders') }} o
JOIN {{ ref('stg_line_items') }} li USING (order_id)
{% if is_incremental() %}
WHERE o.order_date >= (SELECT MAX(order_date) FROM {{ this }})
{% endif %}
GROUP BY 1, 2, 3, 4

dbt-glue: when it actually makes sense

dbt-glue is the right choice in exactly two scenarios: (1) your source data is in formats that Redshift and Athena cannot efficiently read (Delta Lake, Hudi, Iceberg with compaction requirements), or (2) you need Spark's processing model for window functions on multi-terabyte datasets where Athena times out and Redshift Spectrum is too slow. Outside these cases, dbt-glue adds complexity and cost without benefit.

# profiles.yml — dbt-glue
your_project:
  target: prod
  outputs:
    prod:
      type: glue
      role_arn: arn:aws:iam::123456789012:role/GlueDbtRole
      region: us-east-1
      workers: 10
      worker_type: G.1X
      schema: dbt_prod
      database: analytics
      session_provisioning_timeout_in_seconds: 120
      datalake_formats: iceberg   # enables Iceberg writes
      conf: >
        spark.sql.extensions=org.apache.iceberg.spark.extensions.IcebergSparkSessionExtensions
        spark.sql.catalog.glue_catalog=org.apache.iceberg.spark.SparkCatalog
        spark.sql.catalog.glue_catalog.warehouse=s3://your-datalake/warehouse

✓ Decision rule

Use dbt-redshift if data is in Redshift. Use dbt-athena if data is in S3 as Parquet/ORC with Glue Catalog. Use dbt-glue only if you need Iceberg table format or have models that require Spark-specific capabilities (e.g., ML feature computation via Spark MLlib). There is no middle ground worth engineering.
