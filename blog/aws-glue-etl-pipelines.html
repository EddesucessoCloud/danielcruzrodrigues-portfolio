1. What is AWS Glue?

AWS Glue is a fully managed, serverless Extract-Transform-Load (ETL) service. It handles the undifferentiated heavy lifting of provisioning clusters, managing dependencies, and scaling workers — so you focus on business logic, not infrastructure. Under the hood it runs Apache Spark (and Python shell for lightweight jobs), so if you already know PySpark you're immediately productive.

I've used AWS Glue extensively at NTT DATA on the Banco Itaú data platform to build bronze-to-silver and silver-to-gold transformation layers over Delta Lake on S3, with the Data Catalog acting as the central metadata layer shared across Athena, Redshift Spectrum, and EMR.

Key positioning: Glue is not just an ETL runner. The Data Catalog is arguably its most valuable component — a Hive-compatible metastore that decouples schema from storage and powers downstream analytics services.

2. Core Components Deep Dive

Data Catalog

The Data Catalog is a centralised metadata repository compatible with the Apache Hive Metastore. Databases, tables, and schemas are stored here and shared across Glue Jobs, Athena, Redshift Spectrum, and EMR — no per-service schema duplication.

  • Databases — logical namespace groupings (e.g., bronze, silver, gold).
  • Tables — metadata definitions pointing to S3 prefixes, including partition keys, SerDe properties, and column types.
  • Connections — JDBC definitions for RDS, Redshift, and other JDBC-compatible sources, stored securely for job reuse.

Crawlers

Crawlers automatically inspect data stores (S3, JDBC, DynamoDB) and populate the Data Catalog. They detect schema, partitions, and data formats (Parquet, ORC, JSON, CSV, Avro).

Production warning: Crawlers run on a schedule and re-infer schema on every run. For high-cardinality partition tables, this is slow and expensive. Prefer programmatic table registration via the Glue SDK or Terraform for production pipelines — reserve Crawlers for data discovery on new sources.

Glue Studio

A visual drag-and-drop ETL authoring tool. Useful for quick prototyping and for less technical stakeholders to understand pipeline shape. For production code, always use the underlying Python/Scala scripts directly — the generated code adds unnecessary abstraction layers.

Glue Data Quality

A native data quality engine built on the open-source Deequ library. Define rules with DQDL (Data Quality Definition Language) and attach them to ETL jobs or Catalog tables. Failed rules can halt a pipeline or write quarantine records — critical for data SLAs.

3. Job Types: Choosing the Right Runtime

Type Runtime DPU Unit Best For
Spark ETL PySpark / Scala Standard / G.1X / G.2X / G.4X / G.8X Large-scale batch transforms (>1 GB)
Spark Streaming PySpark (micro-batch) G.1X / G.2X Near-real-time S3 / Kinesis ingestion
Python Shell Python 3.9 0.0625 or 1 DPU Lightweight tasks, API calls, small files (<100 MB)
Ray Python (Ray) Z.2X ML inference, distributed Python (non-Spark)
Cost tip: Use G.1X workers (4 vCPU, 16 GB RAM) as default. Only scale to G.2X when you hit memory pressure on wide joins or large aggregations. G.4X/G.8X are rarely needed for standard ETL — they're designed for ML feature engineering at scale.

4. Production PySpark Job Example

The following job reads raw JSON events from an S3 bronze prefix, applies transformations, and writes Parquet to the silver layer with dynamic partition overwrite — a core pattern in medallion architecture pipelines. 

# glue_bronze_to_silver.py
# Reads raw JSON events → writes partitioned Parquet to silver layer
import sys
from awsglue.transforms import *
from awsglue.utils import getResolvedOptions
from pyspark.context import SparkContext
from awsglue.context import GlueContext
from awsglue.job import Job
from pyspark.sql import functions as F
from pyspark.sql.types import StructType, StructField, StringType, LongType, TimestampType

# ── Job init ──────────────────────────────────────────────
args = getResolvedOptions(sys.argv, ['JOB_NAME', 'SOURCE_PATH', 'TARGET_PATH'])
sc  = SparkContext()
gc  = GlueContext(sc)
spark = gc.spark_session
job   = Job(gc)
job.init(args['JOB_NAME'], args)

# ── Explicit schema — never infer on raw data ─────────────
schema = StructType([
    StructField("event_id",   StringType(),    False),
    StructField("user_id",    StringType(),    False),
    StructField("event_type", StringType(),    True),
    StructField("amount",     LongType(),      True),
    StructField("ts",         TimestampType(), False),
])

# ── Read ──────────────────────────────────────────────────
df = (
    spark.read
    .schema(schema)
    .option("multiLine", True)
    .json(args['SOURCE_PATH'])
)

# ── Transform ─────────────────────────────────────────────
df_silver = (
    df
    # Drop malformed rows missing primary keys
    .filter(F.col("event_id").isNotNull() & F.col("user_id").isNotNull())
    # Partition columns — derived from ts for efficient pruning
    .withColumn("year",  F.year("ts"))
    .withColumn("month", F.month("ts"))
    .withColumn("day",   F.dayofmonth("ts"))
    # Normalise event_type to lowercase
    .withColumn("event_type", F.lower(F.col("event_type")))
    # Dedup within the batch on event_id
    .dropDuplicates(["event_id"])
)

# ── Write — dynamic partition overwrite ───────────────────
spark.conf.set("spark.sql.sources.partitionOverwriteMode", "dynamic")

(
    df_silver
    .write
    .mode("overwrite")
    .partitionBy("year", "month", "day")
    .parquet(args['TARGET_PATH'])
)

job.commit()
Why explicit schema? Schema inference reads the entire dataset before processing. On large S3 prefixes this adds minutes and significant cost. Always define the schema explicitly in production jobs.

5. DPU Sizing & Cost Optimisation

DPU (Data Processing Unit) is Glue's billing unit — 4 vCPUs + 16 GB RAM, billed per second with a 1-minute minimum. Getting DPU sizing wrong is the fastest way to overspend on Glue.

Sizing formula

Start with: (input_size_GB × 4) / worker_memory_GB as a baseline, then profile with Glue job metrics.

  • Enable job metrics in the Glue Console — the Memory Profile chart shows heap usage per executor.
  • If max heap stays below 60%, halve the worker count or downgrade worker type.
  • If jobs spill to disk (shuffle write > 0 on small datasets), upgrade to G.2X.

Spark configuration tuning

# Set in job parameters or GlueContext
spark.conf.set("spark.sql.shuffle.partitions", "200")     # default 200, reduce for <10 GB
spark.conf.set("spark.sql.adaptive.enabled", "true")       # AQE — auto-coalesces partitions
spark.conf.set("spark.sql.adaptive.coalescePartitions.enabled", "true")
spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true")  # AQE — handles skewed joins
Adaptive Query Execution (AQE) is a game changer for skewed datasets. It repartitions automatically at runtime. Enable it on any job doing wide joins over financial or user-event data — we cut p99 latency by ~30% on the Itaú pipeline after enabling it.

6. Glue vs EMR vs Lambda vs Step Functions

This is the most common architecture question I get from teams new to AWS data engineering:

Service Best When Avoid When
AWS Glue (Spark) Batch ETL >1 GB, need managed Spark, Data Catalog integration Sub-minute latency required, complex custom Spark libs
EMR on EC2/EKS Long-running clusters, custom Spark builds, cost-sensitive at scale (>100 DPUs) Occasional batch jobs — cold start + cluster cost is wasteful
Lambda Event-driven micro-transforms <100 MB, fan-out file routing Anything with Spark, stateful processing, >15 min runtime
Step Functions Orchestrating multi-service workflows (Glue + Lambda + SNS) Actual data transformation — it's an orchestrator, not a processor
Glue (Python Shell) API ingestion, metadata updates, small file aggregation Anything needing distributed compute

7. Production Best Practices

Job bookmarks

Enable job bookmarks (--job-bookmark-option job-bookmark-enable) on incremental loads. Glue tracks which S3 objects have already been processed using a checkpoint stored in the Glue state backend — no need to manage watermarks manually.

Gotcha: Job bookmarks track S3 objects by path + modification time. If upstream jobs overwrite files with the same path (common in overwrite-mode pipelines), bookmarks silently skip them. Use append-mode writes or disable bookmarks on full-load jobs.

Error handling and retries

# Wrap transformation stages in try/except and write failed records
from pyspark.sql import functions as F

df_valid   = df.filter(F.col("amount") >= 0)
df_invalid = df.filter(F.col("amount") < 0)

# Ship invalid records to a quarantine prefix — never silently drop
df_invalid.write.mode("append").parquet("s3://my-bucket/quarantine/events/")

CloudWatch metrics to monitor

  • glue.driver.aggregate.numFailedTasks — non-zero means data quality issues or OOM.
  • glue.driver.aggregate.shuffleLocalBytesRead — high values indicate partition skew.
  • glue.driver.jvm.heap.usage — consistently >80% means you need bigger workers.
  • glue.ALL.jvm.heap.usage — per-executor view to catch outlier executors.

Security

  • Always attach a Glue security configuration with S3 SSE-KMS encryption for job bookmarks, logs, and temp data.
  • Use VPC networking when reading from RDS or Redshift — never expose JDBC endpoints publicly.
  • Scope IAM roles to specific S3 prefixes, not s3:*/*.

8. Conclusion

AWS Glue is an excellent default choice for batch ETL in a data lake on AWS when your data exceeds a few hundred MB and you want to avoid managing Spark infrastructure. The key to making it production-grade is: explicit schemas, AQE enabled, correctly sized DPUs, job bookmarks for incremental loads, and quarantine logic for invalid records.

Where Glue falls short is ultra-low latency, very custom Spark builds, or extreme cost-sensitivity at scale — at that point EMR Spot clusters or Databricks start to win on economics. But for most batch pipelines in a greenfield AWS environment, Glue + Data Catalog is the fastest path to a maintainable, serverless ETL layer.
